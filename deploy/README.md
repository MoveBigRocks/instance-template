# Move Big Rocks Server Deployment

Canonical production deployment guide for Move Big Rocks, the AI-native service operations platform.

Architecture: systemd + Caddy + PostgreSQL with blue-green app slots.

If you are bringing `movebigrocks.com` live on a standalone host, use
[`Caddyfile.example`](https://github.com/movebigrocks/platform/blob/main/templates/instance-template/deploy/Caddyfile.example)
as the starting point for apex, admin, and api host routing.

## What Gets Installed

| Component | Purpose | Port |
|-----------|---------|------|
| **Caddy** | Reverse proxy + automatic HTTPS | 80, 443 |
| **Move Big Rocks Blue** | App slot A | 8080 (localhost) |
| **Move Big Rocks Green** | App slot B | 8081 (localhost) |
| **Prometheus** | Metrics collection | 9090 (localhost) |
| **Grafana** | Dashboards | 3000 (localhost) |
## Quick Start (Fresh Server)

```bash
# 1) Create a private customer instance repo from the public Move Big Rocks instance template
#    and open that repo in your agent or terminal.

# 2) SSH into Ubuntu 22.04+ host
ssh root@your-server-ip

# 3) Clone the instance repo and run bootstrap
git clone git@github.com:<your-org>/<your-mbr-instance-repo>.git /opt/mbr-instance
cd /opt/mbr-instance/deploy
sudo ./setup.sh

# 4) Fill in mbr.instance.yaml and GitHub secrets in the instance repo.
#    The deploy workflow renders /opt/mbr/.env from those inputs.
#    Manual editing of /opt/mbr/.env should be treated as a break-glass path only.

# 5) Configure deploy SSH key for user 'mbr'
mkdir -p /opt/mbr/.ssh
# append deploy public key to /opt/mbr/.ssh/authorized_keys
chmod 600 /opt/mbr/.ssh/authorized_keys
chown -R mbr:mbr /opt/mbr/.ssh

# 6) Trigger production deploy from the instance repo
```

The deploy workflow will:

- read `mbr.instance.yaml`
- pull the pinned core artifacts
- render `/opt/mbr/.env` from config plus secrets
- render `/opt/mbr/.fleet-config.env` from the disclosed fleet settings in `mbr.instance.yaml`
- run migrations
- deploy to the inactive slot
- health-check it
- switch the active slot

The core source repo is still where Move Big Rocks itself is developed. The customer-facing production path should use a private instance repo as the deployment control plane so core upgrades stay pinned to released artifacts instead of long-lived customer forks.

## Runtime Environment Contract

The runtime environment file at `/opt/mbr/.env` is normally generated by the instance repo workflow.

The deploy workflow renders `/opt/mbr/.fleet-config.env` from `spec.fleet` and installs the `mbr-fleet-heartbeat.service` / `mbr-fleet-heartbeat.timer` units as host-owned assets. The manual `Register Fleet` workflow stores the returned tracking secret at `/opt/mbr/.fleet-secret.env` and enables the timer only when heartbeat is enabled.

If you need to debug it manually, it should look like:

```bash

ENVIRONMENT=production

# Auth
JWT_SECRET=<openssl rand -base64 64>
JWT_EXPIRY=24h
MAGIC_LINK_TTL=15m
COOKIE_DOMAIN=.app.example.com

# Email

FROM_EMAIL=mbr-noreply@example.com
FROM_NAME=Example Support

# Or SMTP
# EMAIL_BACKEND=smtp
# SMTP_HOST=smtp.example.com
# SMTP_PORT=587
# SMTP_USERNAME=<username>
# SMTP_PASSWORD=<password>

# Or SES
# EMAIL_BACKEND=ses
```

The fleet config file generated from `mbr.instance.yaml` looks like:

```bash
MBR_FLEET_API_URL=https://api.movebigrocks.com
MBR_FLEET_INSTANCE_ID=inst_example_prod
MBR_FLEET_INSTANCE_NAME=example-prod
MBR_FLEET_PLATFORM_VERSION=v0.8.1
MBR_FLEET_OPERATOR_EMAIL=owner@example.com
MBR_FLEET_USE_CASE=internal_ops
MBR_FLEET_REGISTRATION_SOURCE=self_hosted
MBR_FLEET_HEARTBEAT_ENABLED=true
```

The host-local secret file is separate and should contain only:

```bash
MBR_FLEET_TRACKING_SECRET=<secret returned by mbr-fleet>
```

## Backup & Restore

PostgreSQL backup, point-in-time recovery, and restore are owned by your
database provider or your PostgreSQL operations runbook. The Move Big Rocks deploy
repo verifies connectivity through `DATABASE_DSN` only.

## Troubleshooting

### Service won't start

```bash
journalctl -u mbr-blue -n 100 --no-pager
journalctl -u mbr-green -n 100 --no-pager
```

### Grafana not loading

1. Check `curl http://127.0.0.1:3000/grafana/api/health` (must use `/grafana/` prefix)
2. Verify Grafana is configured for sub-path serving in `/etc/grafana/grafana.ini`:
   - `root_url = %(protocol)s://%(domain)s:%(http_port)s/grafana/`
   - `serve_from_sub_path = true`
   - `allow_embedding = true` (under `[security]`)
   - `enabled = true` under `[auth.anonymous]`
3. Verify the admin subdomain Caddy block does NOT have a `handle /grafana/*` direct proxy — the app handles Grafana proxying with auth
4. Verify `header /grafana/* { -X-Frame-Options }` is set in the admin Caddy block

### Admin metrics page returns 404

The admin subdomain must NOT have a `handle /metrics { respond 404 }` block in Caddy. This block is correct on public and API subdomains (to prevent external Prometheus scraping), but on the admin subdomain it blocks the app's own `/metrics` page which serves Grafana dashboards behind session auth.

### Database connectivity

```bash
set -a
source /opt/mbr/.env
set +a
psql "$DATABASE_DSN" -c "SELECT current_database(), current_user;"
```

## Security Notes

- Prometheus and Grafana bind to `127.0.0.1` only.
- All external traffic is TLS-terminated in Caddy.
- Keep `/opt/mbr/.env` mode `600` and owned by `mbr`.
